# Agent notes for this repository

This is a Jekyll static single-page app (a chord-identification trainer for
teaching children perfect pitch), deployed to GitHub Pages as a PWA and also
wrapped as an Android app via Capacitor. There is currently no JS bundler and
no framework — familiarity as much as policy. A compile step is acceptable if
proposed on its own merits and it emits build-time artifacts only (never
files to be checked in; see "Generated files policy"). Don't introduce one as
a side effect of an unrelated task, and note the npm machinery that already
exists (`package.json`, `node_modules/`) is for the Android wrapper, not the
frontend.

## Building and testing

Everything goes through `make`:

- `make init` / `make html` / `make serve` — gems, build to `_site/`, dev server
- `make apk-debug` — full Android pipeline (Jekyll → npm ci → generate
  `android/` → `cap sync` → Gradle); needs Node ≥ 22, JDK 21, Android SDK
- `make android-project` / `make android-patches` — see "Generated files" below

(`make` is the orchestrator because it's easy to remember, not because it's
sacred; if the build outgrows it, proposing a replacement is fair game.)

There is no test suite yet. One is actively wanted — it just hasn't been
designed — so treat adding robust tests as welcome, not out of scope. Until
then, verify changes by building and exercising the app
(`make serve` for the web; `docs/android.md` covers the app). The built
output differs from the source: `index.html` and `sw.js` are heavily
templated with Liquid, which generates the chord data, instrument sample
maps, and audio file lists from `_data/*.yml` and `site.static_files` scans.
Judge behavior from `_site/`, not from the source files alone.

## Generated files policy (strict)

Nothing generated gets committed. The model is: commit an input spec
declaring direct dependencies (`package.json`, `Gemfile`,
`capacitor.config.json`), commit the lockfile that resolves it
(`package-lock.json`, `Gemfile.lock`), and let everything derived exist only
in build/cache directories (`_site/`, `node_modules/`, `vendor/`,
`android/`).

The Android native project is the worked example: `android/` is fully
gitignored and regenerated by `scripts/generate_android_project.sh` from the
lockfile-pinned Capacitor template. Our modifications to generated files live
as `patches/android/*.patch`, applied at build time — so what's ours vs. the
template's is exact, and template upgrades fail loudly when a patch stops
applying. Hand-made files that merely land inside the generated tree (icons,
`colors.xml`) live in `android-overlay/`. If you need to change the native
project: `make android-project`, edit `android/` directly, then
`make android-patches` to capture the edit. Never propose committing files
under `android/`.

Corollaries:

- Do not check in build outputs, pre-generated assets (e.g. rendered audio),
  or copies of things a build step can produce. A rejected PR once tried to
  commit pre-generated chord MP3s; the fix was serving the same code paths a
  proper origin instead.
- Vendoring is tolerated only for pinned, never-edited third-party artifacts
  (`js/vendor/Tone.js` is the one instance). If you'd have to *edit* a
  vendored or generated file, that's what the patch mechanism is for.
- Prefer boring, standard orchestration (make, shell, git) over introducing
  new build systems.

## Code style

- JavaScript is plain ES, no classes, no frameworks: `snake_case` functions,
  `SCREAMING_SNAKE_CASE` module-level constants (a leading underscore for
  private-ish state, e.g. `_TONE_STARTED`), 4-space indentation, direct DOM
  manipulation with `onclick=` handlers in the HTML. Match this; don't
  "modernize" it.
- Persistent state goes through `localStorage` (see `STATE_KEY` /
  `SESSION_HISTORY_KEY` in `js/cim.js`).
- Styling is SCSS in `_sass/`, compiled by Jekyll; app data (chords,
  instruments) is YAML in `_data/`, consumed by Liquid — add content there,
  not hardcoded in JS.
- One Jekyll build serves both web and Android. Platform differences are
  gated at *runtime* on `window.Capacitor` (see the service-worker guard in
  `js/cim.js` and the `capacitor-app` class in `index.html`), never via
  separate build flavors or `JEKYLL_ENV` forks.

## Gotchas

- Piano sample filenames contain `#` (e.g. `D#3v11.mp3`), URL-encoded to
  `%23` by the Liquid that builds `INSTRUMENT_INFO`. This is deliberate;
  don't rename the files or "fix" the encoding.
- `sw.js` is a cache-first service worker (web only). If you change which
  assets exist, remember it precaches from Liquid-generated lists; bump the
  cache version (`APP_CACHE`) when changing cached content.
- The `add-android-app` branch is a superseded historical attempt (file://
  WebView + pre-generated audio). Don't copy patterns from it.
- Android releases: pushing a `v*` tag builds a signed APK
  (`versionCode = major*10000 + minor*100 + patch`, so minor/patch < 100 and
  tags must increase). Details in `docs/android.md`.

## Commits and prose

Commit messages explain *how and why*, not just what — especially for
build/infrastructure changes; state verified facts plainly and hedge
inferences. No marketing language ("robust", "seamless", "comprehensive") in
commits, comments, or docs. Keep comments to constraints the code can't
express; don't narrate the obvious.
